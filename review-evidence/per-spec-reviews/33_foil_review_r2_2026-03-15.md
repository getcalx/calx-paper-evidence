## New Blocking Issues

### B1: `_raise_signup_error` double-wraps error detail as JSON-in-JSON (Nolan + Reviewer-DS2)

Line 309: `detail=json.dumps({"detail": error_msg, "error_code": error_code})` passes a JSON string to `HTTPException.detail`. FastAPI already serializes `detail` into the response body. Client receives: `{"detail": "{\"detail\": \"...\", \"error_code\": \"...\"}"}` — a JSON string inside JSON. FE has to `JSON.parse(response.detail)` to extract `error_code`.

**Fix:** Pass a dict directly: `detail={"detail": error_msg, "error_code": error_code}`. One-line change.

### B2: No performance target for complete-signup endpoint (Reviewer-DS2)

The signup flow is the single highest-leverage conversion moment. No acceptance criterion for endpoint latency. The service does 3-4 sequential `pool.acquire()` calls (profile enrichment, get_subscription, start_trial, claim_anonymous_session). Each is a separate connection checkout.

**Fix:** Add AC: `complete-signup` p95 < 500ms. Note the sequential connection pattern as an optimization opportunity (single connection for profile + subscription steps).

## Non-Blocking Observations

### NB1: Session claim counts flatten rich behavioral data (Reviewer-SY4)

`SessionClaimCounts` records three integers. The anonymous session contains the richest zero-bias behavioral signal (which cards engaged, time-on-card, interaction sequence). Enrich `capture_session_migrated` analytics payload with `top_interaction_categories` during build.

### NB2: `signup_method` CHECK constraint is closed-world assumption (Reviewer-SY4)

When Apple OAuth or magic links arrive, CHECK constraint needs migration. Add a comment noting this, or use ENUM type where adding values is `ALTER TYPE ... ADD VALUE` (no table lock).

### NB3: Commentary suppression is the first lever to pull if conversion underperforms (Reviewer-SY4)

Add a Risks note: if capture funnel conversion is low, re-enable commentary before redesigning cards. The config flag makes this easy — the spec just needs to name the hypothesis.

### NB4: Error mapping via string matching is fragile (Reviewer-SY4 + Reviewer-DS2)

`_ERROR_MAP` matches ValueError message substrings. If anyone changes error messages in `agent_session.py`, mapping silently breaks → 500s. Consider typed exception subclasses long-term.

### NB5: Analytics logging is synchronous in request path (Reviewer-DS2)

Three INSERT statements before response returns. Could be deferred if B2's latency target proves tight.

### NB6: Sequential connection pattern (Reviewer-DS2)

Profile enrichment, get_subscription, start_trial, claim_anonymous_session each acquire separate connections. Single-connection refactor would reduce overhead. Non-blocking because all steps are idempotent.
