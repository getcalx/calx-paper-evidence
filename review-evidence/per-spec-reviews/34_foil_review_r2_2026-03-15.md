## New Blocking Issues

### B1: `TIER2_ASPIRATIONAL_SIM_CEILING` and `TIER2_LEVEL_MISMATCH_THRESHOLD` are still inert (Nolan)

Constants are declared in Spencer Config and documented as classification gates, but never enforced in the logic. Classification depends purely on relative level ordering (which level has highest similarity). Without threshold checks, generalist resumes with high similarity across ALL levels get false-positive Tier 2 on marginal differences.

**Fix:** After determining `user_level`, check:
- If embedding data: user's similarity to JD's target level must be below `TIER2_ASPIRATIONAL_SIM_CEILING`
- If substring fallback: mismatch count must meet `TIER2_LEVEL_MISMATCH_THRESHOLD`

### B2: Duplicate `_find_better_fit_role` vector search (Nolan + Reviewer-DS2)

Called once in classification, result discarded (only title survives on `TierClassification`). Called again in endpoint to get `jd_text` and `catalog_id`. Same pgvector scan, redundant ~20ms + connection acquire.

**Fix:** Add `better_fit_jd_text: str | None = None` and `better_fit_catalog_id: UUID | None = None` to `TierClassification`. Classification already has this data — surface it. One vector search, not two.

### B3: Embedding similarity query conflates cluster identity with seniority level (Reviewer-SY4)

**This is the critical issue.** `cluster_vocabulary` embeddings are NOT seniority-stratified — they're cluster-wide competency phrases. The query groups them by `level_signals.seniority_level` via a join through `competency_clusters`. But when a cluster has level_signals at both IC and director, the SAME vocabulary embeddings appear in BOTH groups with identical values. The GROUP BY produces averages that cannot distinguish seniority in embedding space.

The spec claims "compare resume embedding against vocabulary at each seniority level" but the data doesn't support this — vocabulary embeddings have no seniority dimension. A "stakeholder alignment" phrase on a cluster with both IC and director signals contributes identically to both groups.

**Fix options (Spencer decision):**
- (A) Embed `level_signals.signal_phrase` directly and compare resume against those (they ARE seniority-stratified by definition). Requires adding an `embedding` column to `level_signals` and populating during bootstrap.
- (B) Acknowledge the limitation, make substring matching primary (not supplementary), and use embedding similarity only for the better-fit search where it's correctly applied.
- (C) Create seniority-specific vocabulary views by filtering `cluster_vocabulary` to only include phrases from clusters where ALL level_signals are at a single seniority level (seniority-homogeneous clusters).

### B4: JD seniority classification is substring-only — asymmetric with resume (Reviewer-SY4)

Resume seniority uses embeddings (primary) + substring (supplementary). JD seniority uses substring-only. The spec's own argument for embeddings over substrings (Key Decision 1) applies equally to JDs. If JD seniority is misclassified, entire Tier 2 determination is wrong regardless of resume accuracy.

**Fix:** Either embed the JD text and use the same approach, or explicitly justify the asymmetry. The JD text is already available — embedding it costs one more Voyage API call (~100ms, ~$0.0001).

### B5: No calibration strategy for thresholds (Reviewer-SY4)

Thresholds (0.40, 0.55, 3) have zero empirical basis. No precision/recall targets. No mechanism to detect if classification is correct or incorrect. No A/B framework. Thresholds may need to vary per catalog (0.40 might work for SWE but not Marketing).

**Fix:** Add to spec:
- Precision/recall targets for Tier 2 classification
- Manual review queue: sample N classified sessions for human labeling
- Per-catalog threshold logging so domains can be tuned independently
- AC for threshold validation

### B6: Sequential code path must be removed — make `asyncio.gather` canonical (Reviewer-DS2)

The spec presents two code paths: sequential (lines ~774-883) and parallel `asyncio.gather` (lines ~886-921). Both exist. An implementer could build either. The sequential path means better-fit cards arrive after aspirational cards finish PLUS a full second LLM call. Total stream could exceed 12s.

**Fix:** Remove the sequential code path entirely. `asyncio.gather` is the sole canonical implementation.

### B7: 1200ms hold should be FE-side, not server-side `asyncio.sleep` (Reviewer-DS2)

Server-side `await asyncio.sleep(1.2)` holds the SSE connection open doing nothing for 1.2s. Wastes a Railway connection slot. Could trigger FE reconnection/timeout logic on 1.2s of silence.

**Fix:** Emit `tier_section(better_fit)` with `hold_ms: 1200` field. FE implements the delay client-side before rendering the next card. Server emits cards immediately after the separator. Frees server resources.

## Non-Blocking Observations

### NB1: Embedding similarity query semantics are subtler than documented (Nolan)

The query measures "similarity to vocabulary in clusters that happen to have signals at that level" — not "similarity to vocabulary AT that level." Worth documenting actual semantics even if B3 is addressed.

### NB2: Ensemble weighting is unprincipled (Reviewer-SY4)

Embedding is "primary" and substring is "fallback" — they're never combined. If embeddings say IC (0.42) but substring says director (8 matches vs 2), embeddings win. A proper ensemble would weight both signals. Netflix Prize case study: value is in combining orthogonal signals.

### NB3: No Free Lunch across functional areas (Reviewer-SY4)

Same thresholds for all roles. Seniority signals manifest differently: engineering has distinct vocabulary by level, marketing has heavy overlap. Log per-catalog accuracy separately.

### NB4: Substring matching misses morphological variants (Reviewer-SY4 + Nolan)

`if phrase in resume_lower` misses word order/form variations. At minimum use word boundaries (`\b`). Consider token-overlap scoring.

### NB5: `_find_better_fit_role` two-pass search is unnecessary (Reviewer-DS2)

First searches without exclusion, checks result, then re-searches with exclusion. Just search with exclusion from the start.

### NB6: `embedding_similarity` on `Tier2Context` is a leaky abstraction (Reviewer-DS2)

Raw cosine value in the FE contract has no product meaning. Move to structlog metrics. If FE needs confidence, use `match_confidence: "high" | "medium" | "low"`.

### NB7: Resume embed could run in parallel with catalog resolution (Reviewer-DS2)

`embed()` and `get_session_catalog_id()` are independent. `asyncio.gather` would shave ~20ms off pre-LLM overhead.

### NB8: Pre-LLM overhead budget is undercounted (Reviewer-DS2)

Spec estimates ~120ms. Realistic: Voyage API (100-200ms with RTT) + catalog lookup (10-20ms) + classification queries (30-40ms) + better-fit search (15-20ms) = 165-280ms. Add honest breakdown.

## Spencer Decisions (LOCKED — 2026-03-15)

**B3: Substring matching is primary.** Short-phrase embeddings (3-5 words in 1024-dim space) won't discriminate seniority — they'll cluster by competency domain, not level. Reviewer-SY4 recommended Option A (embed level_signals), Reviewer-SY5 recommended Option B (substring primary). Reviewer-SY5's practical concern won: "Built API endpoints" (IC) and "defined API strategy" (director) will embed close together because they share the domain. Ship substring matching, add logging, escalate to LLM-as-a-judge if recall proves insufficient. Embeddings stay for better-fit JD search only (resume-to-JD, correctly applied).

**B4: Substring matching for JDs too.** JDs are formulaic enough that substring matching has high recall. No additional API call needed. Symmetric with B3 decision.
