## Blocking Issues

### B1: `start_trial` silently fails if no `subscriptions` row exists (Nolan)

`start_trial` does `UPDATE subscriptions SET status = 'trialing' ... WHERE user_id = $1 AND status = 'free'`. If no row exists (first-time signup), the UPDATE finds nothing and trial is never started. The user gets free tier instead of 14-day Pro.

**Fix:** Call `get_subscription(pool, user_id)` before `start_trial` to ensure the row exists.

### B2: Trial start timing contradicts merged Spec 25 (Sloane)

Spec 33 calls `start_trial()` at signup. Spec 25 (already merged) explicitly starts trial on first resume upload, not signup. The capture funnel PRD says signup. These contradict.

**Fix:** Spencer decision required. If trial at signup: amend Spec 25. If trial at resume upload: remove `start_trial()` from `complete-signup`. Do not silently override a merged spec.

### B3: `_extract_metadata` double-decodes JWT + uses wrong key for provider (Nolan + Sloane)

`_extract_metadata` imports private `_decode_jwt` from `api/auth.py` and re-decodes the JWT (already decoded by `get_current_user`). Additionally, `metadata.get("iss", "")` is wrong â€” `iss` is the top-level issuer claim, not the provider name. Supabase stores provider in `app_metadata.provider`.

**Fix:** Either (a) extend `AuthenticatedUser` to carry `avatar_url` and `provider`, or (b) store decoded payload on `request.state`. Fix provider extraction to use `app_metadata.provider`.

### B4: JSONB string interpolation in `_log_signup_event` (Nolan)

```python
f'{{"signup_method": "{signup_method}", "session_claimed": {str(session_claimed).lower()}}}'
```

Not parameterized. Use `json.dumps()` instead.

### B5: Analytics event names diverge from PRD (Nolan + Sloane)

PRD specifies four events: `capture_signup_method`, `capture_signup_complete`, `capture_session_migrated`, `capture_reverse_trial_started`. Spec logs one: `signup_complete`. No contract for which events are FE-fired vs BE-fired.

**Fix:** Add analytics events table with event name, trigger, owner (FE/BE), payload shape. Map all four PRD events.

### B6: `full_name` collection for email/password signups unspecified (Nolan)

OAuth users get `full_name` from Google profile. Email/password users get email prefix. Does the frontend collect a name field? If yes, it needs a path to the backend (add `full_name` to `CompleteSignupRequest`). If not, state that email prefix is acceptable.

**Fix:** Spencer decision on name collection strategy for email/password.

### B7: ValueError to HTTP status code mapping is broken (Nolan)

`claim_anonymous_session` raises `ValueError` for all failure cases. The service catches `"already claimed"` but re-raises everything else. A bare `raise ValueError` produces 500, not 409/410/403. No try/except in the endpoint maps messages to HTTP codes.

**Fix:** Add error mapping in endpoint:
- "already claimed by another user" â†’ 409
- "Claim token has expired" â†’ 410
- "Invalid claim token" â†’ 403
- "Session not found" â†’ 404

### B8: Commentary suppression targets wrong file (Nolan)

Spec says implement in `api/catalog.py`. Commentary events are emitted in `agent/pipeline.py` (line 273-279), where suppression already exists. The spec is modifying a file that doesn't contain the relevant code.

**Fix:** Remove `api/catalog.py` from chunk 33C. Verify existing suppression in `pipeline.py`. If the config flag is needed as an override, add it there.

### B9: Missing error response model for frontend (Sloane)

Error codes listed but no response body shape defined. Frontend can't branch on status alone for distinct UX per error.

**Fix:** Define error response model:
```typescript
interface SignupErrorResponse {
  detail: string;
  error_code: "session_claimed" | "claim_expired" | "claim_invalid" | "unauthorized";
}
```

### B10: `profile_created` field is meaningless (Sloane)

Set `True` only when profile query returns `None`, but `get_current_user` auto-creates. Will almost always be `False`. Frontend needs "is this first signup?" â€” derivable from `signup_method IS NULL`.

**Fix:** Rename to `is_first_signup`, set based on whether `signup_method` was null before this call. Or remove and let FE derive.

### B11: Redirect hardcoded to `/canvas` with no session context (Sloane + Reviewer-DS2)

User just completed a translation reveal â†’ clicked save â†’ signed up â†’ lands on blank canvas with no connection to what they experienced. Broken feedback loop.

**Fix:** Make redirect dynamic: `/canvas?session={session_id}` when session was claimed. Or define a post-signup landing that confirms the save.

### B12: No `/signup` page design spec exists (Vivian)

Design spec Section 14 says "Signup flow â€” separate spec." That spec was never written. Faye has nothing to build from: no layout, no OAuth button styling, no error state treatments, no brand application.

**Fix:** Vivian produces `/signup` design section for `design-spec-versed-v1.md`. Spencer approves before Faye scopes.

### B13: Emotional rupture at CTA-to-signup transition (Reviewer-BH6 via Vivian)

User goes from warm, dark, typography-forward translation reveal to an undesigned signup page. Jarring context switch triggers loss of control at the moment of highest engagement.

**Fix:** Spencer decides: modal overlay (preserving Basalt environment + visible translation cards) vs. full page navigation in brand. Vivian designs whichever.

### B14: Error states (409, 410, 403) have no visual treatment (Vivian)

These errors occur at the moment of highest commitment. No recovery paths designed. 410 (expired claim) is especially loaded â€” "You told me it was mine, now it's gone."

**Fix:** Design error states per code. 409 â†’ "Already signed up â€” log in instead." 410 â†’ "Session expired â€” start a new translation (2 min)." 403 â†’ generic fallback. All in Oxidized palette.

## Non-Blocking Issues

### NB2: `UserProfileResponse` Pydantic model missing new fields (Nolan)

### NB3: VERSED_DATA_CONTRACTS.md not updated (Nolan)

### NB4: No rate limiting on `complete-signup` (Nolan)

Idempotent so damage limited, but rate limit prudent.

### NB5: `user_activity_logs` valid event types list needs `signup_complete` (Nolan)

Data contracts enumerate valid types â€” add it.

### NB6: `session_claim_counts` is an untyped dict at rendering boundary (Sloane)

Faye doesn't know what keys to expect. Type explicitly in TS contract or match Spec 23's `ClaimSessionResponse` shape.

### NB7: `session_claim_counts` is a wasted emotional opportunity (Vivian + Reviewer-BH6)

Data flows through but nothing catches it. "3 translations saved to your account" closes the emotional loop. Design a brief welcome confirmation on canvas arrival.

### NB8: No OAuth redirect emotional continuity (Reviewer-BH6 via Sloane)

Full-page redirect to Google breaks context. Frontend should preserve translation state in localStorage before redirect, restore on return.

### NB9: No `display_name` in response (Reviewer-DS2 via Sloane)

Frontend needs a second API call for welcome message. Add `display_name: string | null` to save the round trip.

### NB10: Commentary suppression chunk assignment contradicts spec text (Sloane)

"Not a separate chunk" but assigned to chunk 33C. Clarify.

### NB11: Avatar fallback for email users creates visual class system (Vivian)

OAuth users get photos, email users get nothing. Default: 28px Terracotta circle with first initial.

### NB12: Signup-to-canvas performance target missing (Reviewer-DS2 via Vivian)

No latency target for CTA click â†’ canvas with translations. Propose: < 2000ms OAuth, < 3000ms email/password.

### NB13: 72-hour claim TTL vs emotional urgency mismatch (Reviewer-BH6 via Sloane)

"Save your translation â€” it's yours" but ownership has invisible time limit. 410 error needs low-cost recovery framing, not punishment.

### NB14: `signup_method` usage clarity needed (Reviewer-DS2 via Vivian)

Add one-line note: "analytics-only for v1, no frontend branching."

### NB15: Commentary suppression instrumentation (Reviewer-SY4 via Vivian)

Log `commentary_suppressed: true` on anonymous sessions for future conversion analysis.
